# グラフ/木の探索（Graph Traversal）

「つながり」を扱うデータ構造がグラフ。フォロー関係、カテゴリの親子、依存関係、
経路探索など、サービスの多くの場面に現れる。木はグラフの特殊形（閉路が無く連結）なので、
グラフの探索がそのまま木にも使える。

## グラフの表し方：隣接リスト

グラフは「ノード（頂点）」と「エッジ（辺）」からなる。表現は主に2つ。

- **隣接リスト**：`dict[node] -> [隣接ノード, ...]`。実務ではほぼこれ。
  必要な分だけ持つのでメモリは O(V + E)、隣を列挙するのも速い。
- 隣接行列：V×V の表。密なグラフ向きだが、疎なグラフでは無駄が多い。

無向グラフ（友人関係など、向きが無い）は、辺 `(a, b)` を `a->b` と `b->a` の
**両方向**登録する。有向グラフ（フォロー、依存）は片方向だけ登録する。

## 幅優先探索（BFS）：近いものから

キュー（FIFO）で「今のノードの隣」を全部入れてから次に進む。
start に近いノードから順に広がるので、**重みなしグラフの最短経路（最少ホップ数）**が求まる。

- 「何ホップで到達するか」（SNS の何人でつながるか）→ BFS
- 「最も近い◯◯を探す」→ BFS

`shortest_hops` がこれ。ゴールに初めて到達した時の距離が最短になる。

## 深さ優先探索（DFS）：行けるところまで潜る

「隣へ進めるだけ進み、行き止まりで戻る」。スタック、または**再帰**で書ける。
本実装は再帰版で、再帰そのものの練習にもなる。

- 全ノードを一通り辿る、連結成分を求める → BFS でも DFS でもよい
- 経路の全列挙、閉路検出、トポロジカルソート → DFS 向き

## 必ず要る「訪問済み集合（visited）」

グラフには閉路（ぐるっと戻る道）がありうる。visited が無いと同じノードを
無限に辿り続ける。**一度訪れたノードを集合に記録し、再訪しない**のが鉄則。
集合のメンバー判定は平均 O(1)（→ `data_structures`）なので、探索全体を O(V + E) に保てる。

- BFS は「キューに入れた時点」で visited に登録する（同じノードの重複投入を防ぐ）。
- DFS は「訪れた時点」で登録する。

## 木として使う

木は閉路の無いグラフ。カテゴリの親子、コメントのスレッド、組織図などがこれ。
親→子の隣接リストにすれば、BFS で階層順、DFS で深さ優先に辿れる。
特別なコードは要らず、同じ探索がそのまま使える。

## 設計上の判断とトレードオフ

- **最短ホップが要るなら BFS**。DFS は最短を保証しない。
- **再帰 DFS は深いグラフでスタック超過（RecursionError）**に注意。深い場合は明示スタックの反復版に。
- **visited を必ず持つ**。閉路のあるグラフで忘れると無限ループ。
- **有向か無向か**を最初に決め、辺の登録方向を揃える。ここの取り違えが探索結果を狂わせる。

## 計算量

隣接リストなら、BFS も DFS も各ノード・各辺を一度ずつ見るので **O(V + E)**。
visited による O(1) のメンバー判定が前提。
