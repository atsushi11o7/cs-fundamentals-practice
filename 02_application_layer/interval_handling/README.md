# 区間・範囲の処理（Interval Handling）

予約システム、スケジュール調整、範囲の衝突チェックなどで頻出する「区間の重なり」の扱い。
「2つの区間が重なるか」を正しく書くこと、そして「多数の区間の中に重なりがあるか」を
効率よく調べることの2段階がある。

## 「重なる条件」を直接考えると複雑になる

2区間が重なるパターンは複数ある（片方が中に入る、部分的に被る、など）。
これを全部列挙して条件を書こうとすると、場合分けが増えてバグりやすい。

## 「重ならない条件」を否定するのが定石

重ならないのは、たった2パターンしかない。
「区間1が区間2より完全に前」または「区間1が区間2より完全に後」。
```
重ならない: e1 <= s2  （1が完全に前）  or  e2 <= s1  （1が完全に後）
重なる    : not (e1 <= s2 or e2 <= s1)
```
シンプルな条件の否定を取る、という発想がポイント。

## 境界の扱いは仕様で決める

`[10, 12)` と `[12, 14)` が「重なる」か「接するだけでOK」かは仕様次第。
`<=` と `<` のどちらを使うかで挙動が変わる。実装前に、境界が接する場合の扱いを明確にする。

## 多数の区間：全ペアは O(N²)、ソートで O(N log N)

「リスト内に重なりがあるか」を全ペア比較すると O(N²)。
開始時刻でソートすれば、隣接する区間だけ調べればよくなり O(N log N) に改善できる。
ソート済みなら「今の開始 < 前の終了」で重なりを判定できる。

## 設計上の判断とトレードオフ

- **境界条件を仕様と一致させる**：`<` と `<=` の取り違えが、接触を重なり扱いする／しないの
  バグに直結する。
- **規模に応じて手法を選ぶ**：少数なら全ペアでも十分。大量になるならソートして隣接比較、
  さらに頻繁な問い合わせがあるなら区間木などのデータ構造を検討する。

## 計算量

- 2区間の判定 `overlaps`：O(1)
- N区間の重なり有無 `has_any_overlap`（ソート版）：O(N log N)
- 予約登録 `BookingManager.book`：既存 M 件との比較で O(M)
