# データ構造の使い分け

辞書（集計・検索）、リスト（順序・スタック/キュー）、集合（重複排除・メンバー判定）を、
どの場面でどれを選ぶか。処理を書く前の設計判断であり、計算量を大きく左右する。

## まず「何をしたいか」で選ぶ

データ構造は、やりたい操作の計算量で選ぶ。よく使う対応づけ：

| やりたいこと | 選ぶもの | 主要操作の計算量 |
|---|---|---|
| キーで引く・集計する | 辞書 dict | 参照/更新 平均 O(1) |
| 重複を除く・含むか判定 | 集合 set | メンバー判定 平均 O(1) |
| 順序を保つ・末尾で足し引き | リスト list | append/pop(末尾) O(1)、in は O(n) |
| 両端で足し引き（キュー） | deque | append/popleft O(1) |

## 辞書：キーによる即時アクセス

「キーを見て値を引く/更新する」が平均 O(1)。集計・索引・キャッシュの中心。
詳しくは `aggregation`（集計）と `01_data_layer/caching`（キャッシュ）で扱う。

## 集合：メンバー判定と重複排除

`x in some_set` は平均 O(1)。**リストの `x in list` は O(n)** なので、
「今までに見たか」を大量に問い合わせるなら集合に持ち替えるのが定石。
`has_duplicates` はこれで、リストで書くと O(n²) になる処理を O(n) に保つ。

順序は持たない。順序も欲しいときは「判定は集合・順序はリスト」と役割を分ける
（`dedupe_preserve_order` がこの形）。

## リスト：順序とスタック

順序を保つ基本の器。末尾への `append`/`pop()` は O(1) なので、そのまま**スタック**
（LIFO）になる。一方、**先頭の `pop(0)` や `insert(0, x)` は O(n)**（後続をずらすため）。
「先頭で足し引き」したくなったら deque を疑う。

## deque：先頭・末尾どちらも O(1)

キュー（FIFO）はリストの `pop(0)` だと O(n) になってしまう。
`collections.deque` は両端の操作が O(1) なので、キューには deque を使う（`Queue` 実装）。

## 設計上の判断とトレードオフ

- **メンバー判定が多いならリストではなく集合/辞書**。`in list` の O(n) が積もると効く。
- **キーにできるのはイミュータブルなもの**（文字列・数値・タプル）。リストは不可。
- **順序が要るか**で set と list/dict を分ける（dict は挿入順を保つ）。
- **キューに list.pop(0) を使わない**。deque を選ぶ。
- 集合/辞書は O(1) を「平均」で提供する（ハッシュ衝突で最悪は悪化しうる）が、通常は気にしなくてよい。
