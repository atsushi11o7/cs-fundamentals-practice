# レート制限（Rate Limiting）

一定時間あたりの呼び出し回数を制限し、過剰なアクセスからサービスを守る技術。
公開APIの保護、ログイン試行の抑制、コスト管理などで用いる。

## なぜ必要か

無制限にリクエストを受けると、一部の利用者（あるいは攻撃者）の過剰なアクセスで
サービス全体が不安定になる。回数に上限を設けることで、
一部の偏った使い方が全体に波及するのを防ぐ。

## どの単位で数えるか

「誰からのアクセスか」を区別する軸を決める。

- **APIキー / アカウント単位**：利用者を正確に区別できる。公開APIの基本。
- **IPアドレス単位**：手軽だが、会社や学校の共有IP（NAT）だと多数の正常利用者を巻き込む。

## 代表的なアルゴリズム

- **固定ウィンドウ**：一定時間ごとにカウントをリセットする。単純。
  弱点：ウィンドウの境界前後に集中すると、短時間で実質2倍通ってしまう。
- **スライディングウィンドウ**：直近N秒を滑らせて数える。より正確だが実装が重い。
- **トークンバケット**：一定速度でトークンを補充し、リクエストごとに1つ消費する。
  貯めたトークン分の**バースト（一時的な集中）を許容しつつ、平均レートを制限**できる。
  実務で広く使われる。

## 集中集計と分散集計

- **集中（共通ストレージ、例：Redis）**：全サーバが同じ場所で数えるため正確。
  ただしそこに負荷が集中し、単一障害点にもなる。
- **分散（各サーバが個別に数える）**：速く負荷分散できるが、
  合計がばらけて制限をわずかに超えることがある。

正確さを重視するなら集中が基本。規模が大きければ分散と組み合わせる。

## 制限を超えたときの応答

- 処理せず **HTTP 429（Too Many Requests）** を返す。
- **Retry-After**（いつ再試行してよいか）を添えると、呼び出し側が適切に待てる。
- 繰り返す相手には待ち時間を段階的に延ばす（**指数バックオフ**）。

## 監視と調整

最初に設定した制限値が常に正しいとは限らない。
誰がどれだけ呼んでいるか、制限に何回かかったかを記録し、
正常な利用者を巻き込んでいないか／過剰を許していないかを見て閾値を見直す。

## 設計上の判断とトレードオフ

- **IP単位だけに頼らない**：NAT配下の正常利用者を巻き込むため、APIキー単位が望ましい。
- **正確さと性能のトレードオフ**：集中集計は正確だが負荷集中、分散は速いが誤差が出る。
- **トークン補充の上限を忘れない**：補充量を上限（capacity）で頭打ちにしないと、
  無限に貯まってバースト制限が効かなくなる。
