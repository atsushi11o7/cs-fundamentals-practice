# スケーリングのパターン

アクセスが増えたときの引き出し。ロードバランサ、DBレプリケーション、シャーディング、
メッセージキュー、CDN。どれをどの順で打つかを整理する。

## まず「垂直」か「水平」か

- **垂直スケール（スケールアップ）**：1台をより高性能にする。手軽だが上限があり、単一障害点も残る。
- **水平スケール（スケールアウト）**：台数を増やす。上限が高く冗長性も得られるが、
  「どう分散するか」「状態をどう扱うか」という新しい設計課題が生まれる。

以下は水平スケールを支える定番パターン。

## ロードバランサ：リクエストを複数台へ振り分ける

同じ処理をする複数のサーバ（バックエンド）へ、リクエストを分配する。
最も単純なのが**ラウンドロビン**（順番に回す）。ほかに接続数の少ない台へ送る
least-connections などもある。実装の `RoundRobinBalancer` が前者。

前提として、**各サーバが状態を持たない（ステートレス）**ほど分散は楽になる。
セッション等の状態は、サーバ内ではなく共有ストレージ（DBやキャッシュ）に逃がす。

## DBレプリケーション：読み取りを複数へ

多くのサービスは読み取りが書き込みより圧倒的に多い。
**書き込みは主（primary）1台、読み取りは複製（replica）複数台**に振り分けると、
読み取りをスケールできる。代償として、複製への反映に遅延（レプリケーションラグ）があり、
書いた直後に古い値が読まれることがある（結果整合性）。

## シャーディング：データ自体を分割する

1台に収まらないデータを、キーで複数ノードに分割して保存する。
問題は割り当て方。素朴に **`hash(key) % ノード数`** とすると、ノードを1台足しただけで
余りが総入れ替えになり、**ほぼ全キーが別ノードへ移動**する（キャッシュ全崩壊・大量再配置）。

### コンシステントハッシュ

ノードとキーを同じ「円環」上のハッシュ値に配置し、キーは**円環上で自分以降の最初のノード**に
割り当てる。ノードを増減しても、影響を受けるのは円環上で隣接する範囲だけなので、
移動するキーは**おおむね 1/ノード数**に収まる。各ノードを複数の仮想ノードとして置くと、
割り当ての偏りも減らせる。円環上のノード探索は二分探索（→ `02_application_layer/searching`）。

実装では、3ノード→4ノードの追加で、素朴な modulo が約 750/1000 キーを動かすのに対し、
コンシステントハッシュは約 250/1000 に抑えられることを確認できる。

## メッセージキュー：処理を切り離す

重い処理を同期的にやると応答が遅くなる。**キューに積んで、後ろで別プロセスが消化**すれば、
リクエストはすぐ返せる。急なアクセス増もキューが緩衝材（バッファ）になり、
ピークをならせる。代償は、結果がすぐ返らない（非同期）ことと、キュー自体の運用。

## CDN：ユーザーに近い場所へ

画像・CSS・JS などの静的コンテンツを、世界各地のエッジに複製して配る。
ユーザーに物理的に近い場所から返せるので速く、オリジンサーバの負荷も下げる。

## どの順で打つか

やみくもに増やさず、**計測してボトルネックを特定してから**打つ。おおよその典型順:

1. キャッシュ / CDN で無駄な処理・通信を減らす（→ `01_data_layer/caching`）
2. ロードバランサ + アプリサーバ増設（ステートレス化が前提）
3. DB のリードレプリカで読み取りを分散
4. それでも足りなければシャーディングで書き込み・容量を分割
5. 重い処理はメッセージキューで非同期化

## 設計上の判断とトレードオフ

- **ステートレスにするほど水平分散が楽**。状態は共有ストレージへ。
- **レプリケーションは読み取りをスケールするが遅延を生む**（結果整合性を許せるか）。
- **`% ノード数`は増減に弱い**。再配置を抑えたいならコンシステントハッシュ。
- **キューは応答を速くするが非同期になる**。即時に結果が要る処理には向かない。
- **まず計測**。ボトルネックでない所を増強しても速くならない。

## 実装

`scaling_patterns.py` を参照。`RoundRobinBalancer`（順番振り分け）、`modulo_shard`
（素朴なシャーディング）、`ConsistentHashRing`（コンシステントハッシュ）。
ノード追加時の再配置量を modulo と比較して、後者の利点をテストで確認している。
