# 並行処理（Concurrency）

複数の処理が同じデータを同時に触ると、データが壊れる「競合」が起きる。
なぜ起きるのか、どう防ぐのかを、実際に壊れるコードとともに理解する。

## プロセスとスレッド

- **プロセス**：独立したメモリ空間を持つ。互いのメモリを直接触れない。安全だが通信は重い。
- **スレッド**：同じプロセス内でメモリを共有する。速いが、同じデータを同時に触ると壊れる。

この「メモリ共有」が競合の根本原因になる。

## 競合（レースコンディション）はなぜ起きるか

`count += 1` は一見1動作に見えるが、実際は3手順に分かれる。
```
1. 現在の値を読む      (current = value)
2. 1 を足す           (current + 1)
3. 書き戻す           (value = current + 1)
```
この途中で別のスレッドが割り込み、同じ値を読んでしまうと、片方の更新が上書きで消える。
実装の `UnsafeCounter` はこれが起きうる（環境によっては期待値より小さくなる）。

## ロックで守る

「読んで→足して→書く」の一連を、一度に1つのスレッドしか実行できないようにする。
```python
with self.lock:      # ここに入れるのは同時に1スレッドだけ
    current = self.value
    self.value = current + 1
```
`SafeCounter` はこれにより常に正しい値になる。

## ロックのトレードオフ

- ロック範囲が**広い**：競合は確実に防げるが、待ち時間が増えて遅くなる。
- ロック範囲が**狭い**：速いが、守るべき範囲を外すと競合が残る。
- **CPUが高いのに処理が進まない**：スレッドがロックを取り合い、待ち・再試行にCPUを費やす空回り。
  「CPU使用率が高い＝仕事をしている」とは限らない例。

## デッドロック

複数のロックを、スレッドごとに異なる順序で取ると、
互いが相手の解放を待って永久に止まる。
対策として、ロックを取る順序を全体で統一する、などがある。

## 実サービスでの例

ECの在庫管理：残り1個の商品に2人が同時に注文する。
ロックやトランザクションで、在庫がマイナスにならないよう保証する必要がある。
これはまさにここで扱う競合の問題そのもの。

## 設計上の判断とトレードオフ

- **「1行だから安全」は誤り**：`+=` すら複数手順に分かれ、競合しうる。
- **そもそも共有しない設計も有効**：スレッドごとに独立データを持たせる、
  キューで処理を直列化する、といった方法で競合自体を消せる。
- **並列化のしすぎに注意**：ロック範囲が広いと、並列化しても直列とほぼ変わらない速度になる。
