# Webパフォーマンス

「速い」を測ってから直す。何を優先して最適化するか、その考え方と、
サーバ側でコードとして実装できる定番手法を扱う。

## まず計測する

推測で最適化しない。ユーザー体感に近い指標を測り、ボトルネックを特定してから手を打つ。

- **LCP（Largest Contentful Paint）**：主要コンテンツが表示されるまでの時間。体感速度の代表指標。
- ほかに TTFB（最初のバイトが返るまで）、CLS（レイアウトのずれ）など。

「どこが遅いか」を測らずに手を動かすと、効かない所を直して終わる。計測が最適化の出発点。

## 転送を減らす（サーバ側で実装できる核）

パフォーマンス改善の多くは「無駄な転送・処理を減らす」こと。サーバ側で書ける代表が2つ。

### 条件付きリクエスト（ETag → 304）

リソースの内容から **ETag**（内容の指紋）を計算して返す。次回クライアントは
その ETag を `If-None-Match` で送ってくる。内容が変わっていなければ、サーバは
**304 Not Modified を本文なしで返す**。本文の再送を丸ごと省けるので、
変化の少ないリソースで効く。実装の `handle_conditional_get` がこれ。

（`Cache-Control` / `max-age` で「そもそも問い合わせない」キャッシュと組み合わせる。
キャッシュ自体は `01_data_layer/caching` で扱う。）

### 圧縮（gzip）

テキスト（HTML/CSS/JS/JSON）は繰り返しが多く、**gzip でよく縮む**。
転送バイト数が減るので速くなる。実装の `gzip_compress` / `compression_ratio` で、
繰り返しの多いテキストが大幅に縮む（比 0.0x）ことを確認できる。
逆に短い/ランダムなデータはヘッダ分でむしろ増えることもあるので、小さすぎる応答には掛けない。

## フロント側の定番（考え方）

サーバのコードにはならないが、体感速度に直結する重要な手法。

- **画像の最適化**：適切なサイズ・フォーマット（WebP 等）で配る。画像は転送量の大半を占めがち。
- **遅延読み込み（lazy loading）**：画面外の画像やコンポーネントを、必要になるまで読み込まない。
  初期表示（LCP）を優先し、後回しにできるものは後で。
- **重要リソースの優先**：最初の表示に必要な CSS/JS を先に、それ以外は遅延・非同期で。
- **CDN**：静的コンテンツをユーザーに近いエッジから配る（→ `scaling_patterns`）。

## 設計上の判断とトレードオフ

- **計測してから最適化**：ボトルネックでない所を直しても速くならない。
- **条件付きリクエストは変化の少ないリソースで効く**：頻繁に変わるものには恩恵が薄い。
- **圧縮は大きめのテキストに**：小さい/ランダムなデータは縮まず、CPU だけ使う。
- **初期表示を優先する**：すべてを一度に読み込まず、体感に効くものから出す。

## 実装

`web_performance.py` を参照。`etag_for` / `handle_conditional_get`（304 による本文再送の省略）、
`gzip_compress` / `compression_ratio`（圧縮によるサイズ削減）。サーバ側で実装できる核だけを扱い、
フロント寄りの手法は考え方として整理している。
