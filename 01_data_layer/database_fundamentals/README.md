# データベース基礎

テーブル設計・主キー・外部キーの基本から、インデックスがなぜ検索を速くするのか、
トランザクションとACID、ロックまで。データを正しく・速く保存し取り出すための土台。

ストレージエンジンそのものを自作するのは範囲外なので、ここでは Python 標準ライブラリの
`sqlite3` を使い、本物のDBの挙動を実際に動かして観察する（外部依存なし）。

## 主キーと外部キー

- **主キー（Primary Key）**：行を一意に識別する列。重複と NULL を許さない。
- **外部キー（Foreign Key）**：別テーブルの主キーを参照し、リレーションを表す。
  存在しない親を参照する行（孤児レコード）を防ぎ、**整合性**を守る。

SQLite は既定で外部キー制約を無視するため、`PRAGMA foreign_keys = ON` で有効化する。
有効なら、存在しない `user_id` を持つ記事の挿入は `IntegrityError` で拒否される。

## インデックスがなぜ速いのか

インデックスが無いと、DBは条件に合う行を探すために**全行を順に走査**する（フルスキャン、O(n)）。
インデックスは、対象列の値を並べ替えた別の探索構造（B-tree）を持っておく仕組み。
これにより、目的の値へ二分探索的にたどり着ける（O(log n)）。本の巻末索引と同じ発想。

`EXPLAIN QUERY PLAN` で、DBがそのクエリをどう実行するつもりかを覗ける。

- 索引なし：`SCAN articles` … 全走査
- 索引あり：`SEARCH articles USING INDEX idx_articles_user (user_id=?)` … 索引で絞り込み

実装では、同じクエリに対して索引を張る前後で計画がこの2つに変わることを確認できる。

### インデックスのトレードオフ

- **速くなるのは読み取り**。書き込み（INSERT/UPDATE/DELETE）は索引の更新も伴うので**遅くなる**。
- **容量を消費する**。索引はデータとは別に保持される。
- だから「何にでも張る」は誤り。**検索・結合・並べ替えで頻繁に使う列**に絞って張る。
- UNIQUE 制約を付けた列には索引が自動生成される（重複チェックに索引が要るため）。

## トランザクションとACID

トランザクションは「まとめて全部成功、さもなくば全部無かったことにする」処理の単位。
ACID はその性質：

- **A（原子性）**：途中で失敗したら全て取り消す。中途半端な状態を残さない。
- **C（一貫性）**：制約（主キー・外部キー・UNIQUE 等）を破る状態にはしない。
- **I（独立性）**：並行するトランザクション同士が互いの途中経過を見ない。
- **D（永続性）**：commit したら、障害が起きても結果は失われない。

実装の `add_user_with_articles` は、ユーザー1件と記事複数件を1トランザクションで入れる。
`with conn:` が境界で、正常終了なら commit・例外なら rollback する。
途中で失敗させると、**先に入れたユーザーごとロールバックされ、1行も残らない**（原子性）。

## ロックと独立性

複数の処理が同じデータを同時に更新すると、互いの変更を上書きし合う競合が起きる。
DBは**ロック**で、あるトランザクションが触っている間、他の書き込みを待たせて防ぐ。

- ロックの粒度が粗い（テーブル全体）ほど安全だが並行性が下がる。細かい（行単位）ほど速いが複雑。
- ロック待ちが互いを塞ぐと**デッドロック**になる。DBは検知して片方を中断する。
- 「どこまで他の途中経過を見てよいか」を決めるのが**分離レベル**（Read Committed / Serializable 等）。
  厳しくするほど正しいが遅い。速度と正しさのトレードオフ。

（並行制御そのものは `04_reliability_and_scale/concurrency` でコード付きで扱う。）

## 設計上の判断とトレードオフ

- **索引は読み取りを速く・書き込みを遅くする**。読み多い列に絞る。
- **制約はアプリではなくDBに持たせる**と、どの経路から書いても整合性が守られる。
- **トランザクション境界を正しく引く**。「一緒に成功/失敗すべき操作」を1単位にまとめる。
- **分離レベルは強いほど遅い**。要件に必要な最小限を選ぶ。
