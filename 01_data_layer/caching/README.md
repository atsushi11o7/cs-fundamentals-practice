# キャッシュ

よく使うデータを高速な場所に一時保存し、取得を速くする技術。
速度を得る代わりに「元データとの整合性」という新しい課題を抱える。その扱いが設計の本質。

## なぜキャッシュが必要か

データベースへのアクセスは、メモリ上のアクセスに比べて遅い。
同じデータを何度も読むなら、一度取得した結果を手元に置いておけば、
2回目以降はデータベースに行かずに済む。これがキャッシュの基本的な発想。
読み取りが多く、更新が少ないデータほど効果が大きい。

## 最初の設計判断：整合性をどこまで求めるか

キャッシュを入れると、元データとキャッシュがズレる瞬間が生まれる。
どこまで許すかを最初に決める。

- **即時反映を優先**：変わったら常に最新。正しいが、頻繁に取りに行くので負荷が高い。
- **遅延を許容**：短時間なら古くてよい。負荷は軽いが一時的にズレる。

データの性質で決める。購入数量のように判断に直結するものは正確さ寄り、
閲覧数のように多少ズレても困らないものは遅延許容、というように。

## Cache-Aside パターン

最も基本的な読み取り戦略。アプリケーションがキャッシュとDBを仲介する。

1. まずキャッシュを見る
2. あれば、それを返す（ヒット）
3. なければDBから取得し、キャッシュに保存してから返す（ミス）

実装の `get_user` がこの流れ。「キャッシュ → 無ければDB → 保存」を必ずセットで書く。

## TTL（有効期限）

各エントリに「いつまで有効か」を持たせ、期限が切れたら破棄する。

- 長いTTL：負荷は減るが、古い値が出やすい
- 短いTTL：新しい値を保ちやすいが、負荷が増える

正確さを重視するなら短めにする。実装では保存時刻を記録し、経過時間で判定している。

## 無効化（Invalidation）

TTLだけに頼ると、期限が切れるまで古い値が出続ける。
これを防ぐのが無効化：**元データを更新したら、対応するキャッシュを即座に消す**。
次の読み取りでDBから取り直され、最新が反映される。実装の `update_user` がこれ。

> キャッシュの無効化はバグの温床として知られる。更新処理を書くときは、
> 「この更新でどのキャッシュが古くなるか」を必ずセットで考える習慣が要る。

## キャッシュキーの設計

何のデータかを一意に識別できるキーにする。
ユーザーごとに異なるデータなら `cart:count:{userID}` のようにIDを含め、
他のユーザーのデータと混ざらないようにする。

## 設計上の判断とトレードオフ

- **何でもキャッシュしない**：更新が多いデータはすぐ無効化され、キャッシュの意味が薄い。
  読み多く・更新少ないデータに向く。
- **キャッシュ消失への備え**：キャッシュが一斉に切れると、大量のリクエストが同時にDBへ流れる
  （サンダリングハード）。TTLを少しずつずらす等で緩和する。
- **速度と整合性は交換関係**：速くするほどズレやすい。何を優先するかを明示して設計する。

## 実装

`caching.py` を参照。TTL付きの `SimpleCache` と、Cache-Aside＋無効化を実装した `UserRepository`。
`UserRepository` はDBアクセス回数を数えており、キャッシュヒットで実際にDBアクセスが
減ることをテストで確認できる。
