# データモデリング

サービスの仕様（どう振る舞うか）を、データの構造（どんなテーブルを持つか）に翻訳する作業。
アプリケーション設計の出発点であり、ここでの判断が後続の実装のしやすさを大きく左右する。

## なぜデータモデリングが重要か

アプリケーションは突き詰めれば「データを保存し、取り出し、加工して返す」もの。
そのデータをどう構造化するかが決まらなければ、APIも処理も書けない。
逆に、データ構造が素直に設計されていれば、多くの機能は自然に実装できる。

## 落とし込みの手順

### 1. エンティティを洗い出す
サービスに登場する「モノ」を名詞で挙げる。これがテーブルの候補になる。
例（ブログ）：ユーザー、記事、コメント、タグ。

### 2. 属性を決める
各エンティティが持つデータを列挙する。
例（記事）：id、著者、タイトル、本文、作成日時。

### 3. リレーション（関係）を決める
エンティティ同士のつながりを「1対多」「多対多」で捉える。

- **1対多**：ユーザー1人が記事を複数持つ。
  → 「多」側（記事）に「1」側のID（user_id）を外部キーとして持たせる。
- **多対多**：記事は複数タグを持ち、タグは複数記事に付く。
  → **中間テーブル**（article_tags）を作り、両者のIDのペアを並べる。

### 4. 正規化する
同じ情報を複数の場所に重複して持たないよう整理する。
例：記事テーブルに著者名を直接書くと、著者が改名したとき全記事を書き換える必要がある。
→ 著者は users テーブルに1件だけ持ち、記事は user_id で参照する。
これで著者情報の変更が1箇所で済む。

## テーブル設計の例（ブログ）

```
users
- id (主キー)
- name
- email

articles
- id (主キー)
- user_id (外部キー → users.id)   # 1対多：著者
- title
- body
- created_at

tags
- id (主キー)
- name

article_tags                       # 多対多の中間テーブル
- article_id (外部キー → articles.id)
- tag_id (外部キー → tags.id)
```

## 属性を持つ中間テーブル（関係そのものにデータがある）

`article_tags` はペアをつなぐだけで、中間テーブル自身に属性が無かった。
だが現実の多対多は、**関係そのものが属性を持つ**ことが多い。ECの注文がその典型。

「注文」と「商品」は多対多（1注文に複数商品、1商品は複数注文に登場）だが、
「その注文でその商品を**いくつ**・**いくらで**買ったか」は、注文にも商品にも属さず、
**関係（注文明細）の属性**になる。

```
orders
- id (主キー)
- customer_id (外部キー → customers.id)

products
- id (主キー)
- name
- price                            # 現在の販売価格

order_items                        # 属性を持つ中間テーブル（注文明細）
- order_id  (外部キー → orders.id)
- product_id (外部キー → products.id)
- quantity                         # ← 関係の属性
- unit_price                       # ← 関係の属性（注文時点の単価スナップショット）
```

中間テーブルに `quantity` / `unit_price` を持たせるのがポイント。
「多対多だから中間テーブル」だけでなく、**そこに載る属性を見極める**のが設計の肝になる。

### 参照ではなくコピーする判断（スナップショット）

`unit_price` を `products.price` の外部キー参照にせず、注文時点の値を**明細にコピー**する。
これは一見「重複を持つ＝非正規化」だが、正しい設計。理由は、商品価格は将来変わりうる一方、
**過去の注文金額は変わってはいけない**から。参照にすると値上げで過去の売上が書き換わってしまう。

「重複を消す（正規化）」と「事実を確定させる（スナップショット）」は別の要求。
時間で変わる値を"その時点の事実"として残したいなら、あえてコピーして持つ。

## 正規化のやりすぎ／あえて重複させる判断

正規化の基本は「同じ事実を1箇所だけに持つ」。だが常に最善ではない。

- **悪い重複（正規化で消す）**：`articles` に `author_name` を直接持つ。
  著者が改名すると全記事を書き換える必要があり、書き換え漏れで不整合が起きる。
  → 著者名は `users` に1件だけ持ち、記事は `user_id` で参照する。
- **良い重複（あえて残す）**：上の `unit_price` のように、**時点の事実**を確定保存したい場合。
  あるいは読み取りが極端に多く、JOIN を避けたい場合の非正規化（整合性より性能を優先する判断）。

判断基準は「**その重複は、元が変わったとき一緒に変わるべきか**」。
一緒に変わるべき（＝同じ事実）なら正規化して1箇所に。
変わってはいけない（＝別の事実／時点の記録）なら、あえて分けて持つ。

## 設計上の判断とトレードオフ

- **1対多は「多」側に外部キーを置く**のが基本。「どちらが複数を持つか」で向きを決める。
- **多対多は中間テーブルでしか表現できない**。片方に配列的に持たせようとすると破綻する。
- **正規化はやりすぎない**：厳密に分割しすぎると結合（JOIN）が増えて読み取りが遅くなることがある。
  読み取り性能のために、あえて重複を許す（非正規化）判断もある。整合性と性能のトレードオフ。
- **将来の変更を見据える**：増えそうな属性は、拡張しやすい形（例：可変な属性をJSON型カラムに逃がす）
  にしておくと、後のスキーマ変更を減らせる。

## 補助コード

- `schema_example.py` — ブログのスキーマを dataclass で表現した最小例。
  「1対多は多側に外部キー」「多対多は中間テーブル」という構造そのものと、
  リレーションをたどるメソッドを確認できる。
- `order_schema.py` — 属性を持つ中間エンティティ（注文明細）の例。
  中間テーブルが quantity / unit_price を持つこと、単価をスナップショットとして確定保存し、
  後から商品価格が変わっても過去の注文金額が変わらないことをコードで示す。

いずれも DB は使わず、構造と設計判断そのものをコードで確認できるようにしている。
