# データモデリング

サービスの仕様（どう振る舞うか）を、データの構造（どんなテーブルを持つか）に翻訳する作業。
アプリケーション設計の出発点であり、ここでの判断が後続の実装のしやすさを大きく左右する。

## なぜデータモデリングが重要か

アプリケーションは突き詰めれば「データを保存し、取り出し、加工して返す」もの。
そのデータをどう構造化するかが決まらなければ、APIも処理も書けない。
逆に、データ構造が素直に設計されていれば、多くの機能は自然に実装できる。

## 落とし込みの手順

### 1. エンティティを洗い出す
サービスに登場する「モノ」を名詞で挙げる。これがテーブルの候補になる。
例（ブログ）：ユーザー、記事、コメント、タグ。

### 2. 属性を決める
各エンティティが持つデータを列挙する。
例（記事）：id、著者、タイトル、本文、作成日時。

### 3. リレーション（関係）を決める
エンティティ同士のつながりを「1対多」「多対多」で捉える。

- **1対多**：ユーザー1人が記事を複数持つ。
  → 「多」側（記事）に「1」側のID（user_id）を外部キーとして持たせる。
- **多対多**：記事は複数タグを持ち、タグは複数記事に付く。
  → **中間テーブル**（article_tags）を作り、両者のIDのペアを並べる。

### 4. 正規化する
同じ情報を複数の場所に重複して持たないよう整理する。
例：記事テーブルに著者名を直接書くと、著者が改名したとき全記事を書き換える必要がある。
→ 著者は users テーブルに1件だけ持ち、記事は user_id で参照する。
これで著者情報の変更が1箇所で済む。

## テーブル設計の例（ブログ）

```
users
- id (主キー)
- name
- email

articles
- id (主キー)
- user_id (外部キー → users.id)   # 1対多：著者
- title
- body
- created_at

tags
- id (主キー)
- name

article_tags                       # 多対多の中間テーブル
- article_id (外部キー → articles.id)
- tag_id (外部キー → tags.id)
```

## 設計上の判断とトレードオフ

- **1対多は「多」側に外部キーを置く**のが基本。「どちらが複数を持つか」で向きを決める。
- **多対多は中間テーブルでしか表現できない**。片方に配列的に持たせようとすると破綻する。
- **正規化はやりすぎない**：厳密に分割しすぎると結合（JOIN）が増えて読み取りが遅くなることがある。
  読み取り性能のために、あえて重複を許す（非正規化）判断もある。整合性と性能のトレードオフ。
- **将来の変更を見据える**：増えそうな属性は、拡張しやすい形（例：可変な属性をJSON型カラムに逃がす）
  にしておくと、後のスキーマ変更を減らせる。

`schema_example.py` に、上記スキーマを Python の dataclass で表現した最小例を置く。
