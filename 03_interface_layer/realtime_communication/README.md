# リアルタイム通信

サーバ側で起きた更新を、クライアントへ「すぐ」届けたい場面がある（チャット、通知、
株価、共同編集など）。通常の HTTP は「クライアントが訊いてから答える」ので、
更新を能動的に押し出す仕組みが要る。代表的な3方式と使い分けを扱う。

## なぜ普通のリクエストでは足りないか

HTTP は要求→応答の一往復で終わる。更新をすぐ知りたいクライアントが
短い間隔で何度も訊きに行く（短ポーリング）と、更新が無い時も通信が発生して無駄が多く、
更新から通知までに最大で間隔ぶんの遅れが出る。これを改善するのが以下の方式。

## ロングポーリング

クライアントが訊きに行くのは同じだが、**サーバは新着が来るまで応答を保留**する。
新着が出た瞬間に返すので、短ポーリングより無駄と遅延が減る。特別な仕組みが要らず、
プロキシやファイアウォールも素通りしやすいのが利点。

核心は「サーバが状態を持ち、クライアントが**カーソル（既読位置）以降**を取りに来る」こと。
実装の `MessageLog.poll(cursor)` がこれで、cursor 以降の新着と次のカーソルを返す。
このオフセット管理が**取りこぼし防止**の鍵になる（受け取った位置を必ず更新する）。

## SSE（Server-Sent Events）

サーバ→クライアントの**一方向**ストリーム。1本の HTTP 接続を開いたまま、
サーバがテキストのメッセージを次々と流す。ブラウザが自動再接続してくれるのも特徴。

メッセージは `field: value` の行を並べ、**空行で1メッセージの終端**を表す単純なテキスト。
実装の `format_sse` がこの整形を行う（`data:` 行、`event:`、`id:`、複数行 data の扱い）。
サーバからの通知だけで足りる（クライアントからの送信が要らない）用途に向く。

## WebSocket

**双方向**の常時接続。最初に HTTP で接続し、`Upgrade` ヘッダで WebSocket へ「昇格」する。
以後は軽量なフレームで双方向にやり取りできる。チャットやゲームなど、双方向・低遅延が要る用途向き。

握手では、クライアントの `Sec-WebSocket-Key` に固定 GUID を連結して SHA-1 → base64 した値を
サーバが `Sec-WebSocket-Accept` として返す。これで「WebSocket を理解するサーバだ」と示し、
接続が確立する。実装の `websocket_accept` がこの計算（RFC 6455 の既知の例値と一致することを
テストで確認）。フレーミング（データの区切り方）はここでは扱わず、握手の核だけを示す。

## 使い分け

| 方式 | 方向 | 用途 | コスト |
|---|---|---|---|
| ロングポーリング | 実質サーバ→クライアント | 手軽・互換性重視 | 接続の張り直しが要る |
| SSE | サーバ→クライアント（一方向） | 通知・フィード配信 | 双方向は不可 |
| WebSocket | 双方向 | チャット・ゲーム・共同編集 | 実装/運用がやや重い |

## 設計上の判断とトレードオフ

- **双方向が要るか**：クライアントからも送るなら WebSocket。受け取るだけなら SSE。
- **互換性・単純さ重視ならロングポーリング**：特別なプロトコルが要らず、プロキシに強い。
- **取りこぼし防止**：カーソル / 最後の id を持ち、再接続時にそこから再開する。
- **順序保証と再接続**：切断は必ず起きる前提で、再接続後に欠落なく続きから受け取れる設計にする。

## 実装

`realtime.py` を参照。`MessageLog`（ロングポーリングのカーソル取得）／
`format_sse`（SSE メッセージ整形）／`websocket_accept`（WebSocket ハンドシェイクの
accept キー計算）。実ネットワークは使わず、各方式の核だけを取り出して確認する。
