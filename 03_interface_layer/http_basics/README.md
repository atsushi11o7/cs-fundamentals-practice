# HTTPの基礎

クライアントとサーバが HTTP で何をやり取りしているか。
リクエスト / レスポンス、メソッド、ステータスコードの意味を、
「実際に流れているテキスト」を自分で解析・生成して理解する。

## HTTP はテキストのやり取り

HTTP メッセージは、決まった形のテキスト。リクエストもレスポンスも構造は同じ。

```
<開始行>\r\n            ← リクエスト行 or ステータス行
<ヘッダ名>: <値>\r\n
...\r\n
\r\n                   ← 空行（ヘッダとボディの境界）
<ボディ>
```

改行は `\r\n`（CRLF）。ヘッダとボディの境界は**空行**（`\r\n\r\n`）。
この「空行で分ける」がパースの起点になる。

## リクエスト行：何を・どうする

```
GET /users/1?verbose=true HTTP/1.1
└┬┘ └──────┬──────────┘ └───┬──┘
method     target(パス+クエリ)  version
```

- **method**：操作の種類（GET/POST/…）。「どうするか」。
- **target**：`パス?クエリ`。「何を」。パスがリソースの場所、クエリが絞り込み条件。
- **version**：HTTP のバージョン。

## メソッド：操作の種類

| メソッド | 意味 | 副作用 | べき等 |
|---|---|---|---|
| GET | 取得 | なし | あり |
| POST | 作成 | あり | なし |
| PUT | 置換更新 | あり | あり |
| PATCH | 部分更新 | あり | 場合による |
| DELETE | 削除 | あり | あり |

「べき等（idempotent）」は、同じ操作を何度送っても結果が同じという性質。
再送・リトライで壊れないかに関わる（詳しくは `rest_api_design`）。

## ステータスコード：結果の種類

先頭の数字で大分類が分かる。

- **2xx 成功**：200 OK / 201 Created / 204 No Content
- **4xx クライアント側の誤り**：400 Bad Request / 401 Unauthorized / 403 Forbidden /
  404 Not Found / 405 Method Not Allowed / 429 Too Many Requests
- **5xx サーバ側の異常**：500 Internal Server Error

「誰の問題か（4xx=送り手 / 5xx=受け手）」がコードで伝わるのが要点。

## ヘッダの扱い

- **ヘッダ名は大小文字を区別しない**（`Content-Type` と `content-type` は同じ）。
  実装では小文字に正規化して引けるようにしておくと安全。
- **Content-Length** はボディのバイト数。受信側はこれで本文の終端を知る。
  マルチバイト文字は「文字数」ではなく「UTF-8 のバイト数」で数える点に注意。

## 設計上の判断とトレードオフ

- **境界は空行**：ヘッダとボディの分割は最初の `\r\n\r\n` で行う。
- **Content-Length を正しく**：長さがずれると本文が切れる／余る。自動計算に寄せると安全。
- **ヘッダ名は正規化**：大小の揺れで取りこぼさないよう小文字化して扱う。
- **クエリとパスを分ける**：`?` で割り、パス（場所）とクエリ（条件）を別に扱う。

## 実装

`http_basics.py` を参照。`parse_request`（生リクエスト文字列 → method/path/query/headers/body）、
`build_response`（ステータス/ヘッダ/ボディ → 生レスポンス文字列、Content-Length 自動付与）。
`http.server` 等に頼らず、プロトコルの文字列そのものを扱う。
