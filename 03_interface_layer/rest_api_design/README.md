# REST API設計

サービスの機能を「リソースへの操作」として、URLとHTTPメソッドで表現する設計スタイル。
外部にアプリケーションの機能を公開する際の、標準的な考え方。

## リソース指向で考える

URLは「動詞」ではなく「名詞（リソース）」で表す。操作はHTTPメソッドで表現する。

- 避けたい形：`/getUser?id=1`, `/createUser`
- 望ましい形：`GET /users/1`, `POST /users`

「何を」をURLで、「どうする」をメソッドで、と役割を分けるのがRESTの発想。

## HTTPメソッドの使い分け

| メソッド | 意味 | 例 |
|---|---|---|
| GET | 取得（副作用なし） | `GET /users/1`、`GET /users` |
| POST | 新規作成 | `POST /users` |
| PUT | 置き換え更新（全体） | `PUT /users/1` |
| PATCH | 部分更新 | `PATCH /users/1` |
| DELETE | 削除 | `DELETE /users/1` |

## 階層で関係を表す

リソースの従属関係はパスの階層で表す。
- `GET /users/1/articles` — ユーザー1の記事一覧
- `POST /articles/10/comments` — 記事10へのコメント作成

## ステータスコードで結果を伝える

- 200 OK（成功）、201 Created（作成成功）
- 400 Bad Request（入力不正）、401/403（認証・権限）、404 Not Found
- 429 Too Many Requests、500 Internal Server Error（サーバ側の異常）

## 設計例（ブログAPI）

```
GET    /articles              記事一覧（ページネーション付き）
POST   /articles              記事作成
GET    /articles/{id}         記事1件取得
PUT    /articles/{id}         記事更新
DELETE /articles/{id}         記事削除
GET    /articles/{id}/comments   その記事のコメント一覧
POST   /articles/{id}/comments   コメント投稿
```

## 設計上の判断とトレードオフ

- **動詞をURLに入れない**：操作はメソッドで表す。URLはリソースの場所を示す。
- **GETに副作用を持たせない**：GETでデータを変更してはいけない（キャッシュや再試行で問題になる）。
- **べき等性を意識する**：同じ PUT / DELETE を複数回送っても結果が同じになるよう設計する。
  POST（作成）は繰り返すと重複が生まれうるので、その扱いを決めておく。
- **一貫性を保つ**：命名・階層・レスポンスの形をAPI全体で統一する。
- **大量データにはページネーション**：一覧取得は `?page=2&limit=20` のように分割して返す。

## 練習方法

身近なサービス（TODO、ブログ、ブックマーク等）のAPIを、上記の形で自分で設計してみる。
既存の公開APIのエンドポイント一覧を眺め、「なぜこう切っているか」を考えるのも理解を深める。

## 実装

`rest_api.py` を参照。Web フレームワークを使わず、`(メソッド, パスパターン)` をハンドラに
振り分けるだけの最小ルータ `Router` を自作し、`/users` の CRUD を配線している。
設計の考え方が実際の挙動としてどう現れるかを確認できる。

- **リソース指向のパス** — `/users`（コレクション）と `/users/{id}`（要素）を分ける
- **ステータスコード** — 作成は 201、取得は 200、削除は 204、無ければ 404
- **メソッドの振り分け** — パスは在るがメソッドが無ければ 405 Method Not Allowed
- **べき等性** — 同じ `PUT` を繰り返しても状態は不変。一方 `POST` は繰り返すと
  別リソースが増える（べき等でない）。テストで両者の違いを確認している

ネットワークや DB は使わず、メモリ上の dict をストアにしている。
