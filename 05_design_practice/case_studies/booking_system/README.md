# ケーススタディ（ロードマップ）：予約システム

会議室のような**リソースを時間帯で予約する**サービスを、ゼロから段階的に作る。
「最初の一番単純な形」から始め、要件が広がるたびに設計とコードを育てていく道のり
（ロードマップ）を追う。各段階は動くコード（`stageN.py`）とテストを持つ。

`design_method` の型を、実際に手を動かしながらたどる回。
「まず簡単に作る → 足りなくなった所で足す」を意図的に見せる。

## design_method の型との対応

このケースは型を**ロードマップ形式**でたどる（url_shortener は設計書形式）。
段階を追うごとに、型のステップを一巡しながら対象が広がっていく。

| 型のステップ | 主に扱う段階 |
|---|---|
| 1. 要件確認 | 各 Stage 冒頭の「要件／追加要件」 |
| 2. 機能洗い出し | 各 Stage の機能（book / cancel / list） |
| 3. データ設計 | Stage 2（Booking エンティティを起こす） |
| 4. 構成 | Stage 3（API・エンドポイント） |
| 5. スケール対応 | 「さらに先」節（シャーディング・ロック範囲） |
| 6. 運用監視 | 「さらに先」節（成功率・409率・キャンセル率） |

## この題材で学べること

守りたい不変条件は「**同じリソースで時間帯が重ならない**」こと。
希少資源が「時間帯（連続した区間）」である点が、EC在庫（希少資源が個数）との違い。
区間の重なり判定と、二重予約を防ぐ並行制御が中心になる。

---

## Stage 1：単一リソースの予約（一番単純な形）

*型: 要件確認 → 機能洗い出し → データ設計（最小）→ 構成（動く核）*

**要件（最小）**：1つの部屋に、時間帯 [start, end) の予約を受け付ける。重なれば断る。

- 一番単純に、予約を `(start, end)` のリストで持つ。
- 「重なるか」は自分で場合分けせず、**重ならない条件の否定**で書く
  （→ `02_application_layer/interval_handling` の `overlaps` を再利用）。
- 割り切り：リソースは1つ、単一スレッド前提。

→ `stage1_single_resource.py`（`RoomCalendar`）。まず動く核を作る。ここが議論と拡張の出発点。

## Stage 2：複数リソースとデータモデル（要件が広がる）

*型: 要件確認（追加）→ 機能洗い出し → **データ設計**（この段の主眼）*

**追加要件**：部屋は複数ある。予約はキャンセルできる。部屋ごとの一覧が欲しい。

ここで stage1 の「タプルのリスト」では足りなくなる。「どの部屋の・どの予約か」を
一意に指す必要が出たので、**データモデルを起こす**。

- `Booking(id, resource_id, start, end)` というエンティティにする。
- `id` を振るのは、キャンセル時に「その1件」を一意に指すため（→ 主キーの発想）。
- 重なり判定は**同じ `resource_id` の中だけ**で行う。別の部屋は同時刻でも独立。

→ `stage2_multi_resource.py`（`BookingService`）。データ構造が育つと、機能追加が自然に載る。
「データ設計が心臓部」を体感する段。

## Stage 3：並行アクセスと API（本番に近づける）

*型: **構成**（API）＋ **スケール対応**の入り口（並行制御）*

**追加要件**：同時に複数のリクエストが来る。外部から使える API にする。

stage2 の `book()` は「重なりを確認 → 追加」の2手順。この間に別リクエストが割り込むと、
両方が『空いている』と判断して**二重予約**が生まれる（レースコンディション。
→ `04_reliability_and_scale/concurrency` の check-then-act の競合そのもの）。

- 対策：確認〜追加を**ロックで囲み**、同時に1つしか通さない（`SafeBookingService`）。
  stage2 からの変更は「ロックを足す」ことだけ。
- **API 化**：`03_interface_layer/rest_api_design` の `Router` で外部インターフェースにする。
  ```
  POST   /bookings                 → 201 {id} / 409（重なり）/ 400
  DELETE /bookings/{id}            → 204 / 404
  GET    /resources/{id}/bookings  → 200 [...]
  ```
  重なりは **409 Conflict** で返す（「今は競合している」を意味するコード）。

→ `stage3_concurrency_api.py`（`SafeBookingService` / `BookingApp`）。20スレッドが同じ枠を同時に狙っても、
成功は1件だけ（二重予約が起きない）ことをテストで確認する。

## さらに先（型: スケール対応・運用監視）

- **予約の「仮確保 → 確定」**：席を一時的に押さえて数分以内に確定させる（TTL 付きの hold）。
  → キャッシュの TTL と同じ発想（`01_data_layer/caching`）。
- **多数の予約での重なり判定**：1リソースの予約が大量になれば、全件走査 O(N) ではなく
  開始時刻でソートして隣接だけ見る／区間木を使う（→ `interval_handling` の後半）。
- **スケール**：リソース単位で分割（シャーディング）し、ロックの範囲を1リソースに閉じる
  （→ `scaling_patterns`）。ロック範囲が広いと並行性が落ちる、のトレードオフ。
- **永続化と本番の並行制御**：いまは保存もロックもメモリ／プロセス内。実運用では予約を DB に
  永続化し、二重予約の防止も **DB のトランザクション＋行ロック／制約**で行う。プロセス内ロックは
  アプリサーバを複数台に増やすと跨いで効かない（各プロセスが別のロックを持つ）ため、
  共有の場所（DB）に寄せる必要がある。データ設計と DB での不変条件の守り方は、
  次のケース（EC在庫）で詳しく扱う。
- **運用監視**：予約成功率・競合（409）率・キャンセル率を計測して、枠設計を見直す。

## 進め方の要点（この回のねらい）

- **まず簡単に作って動かす**。最初から完璧を目指さない（stage1）。
- **足りなくなった所で足す**。要件が広がった瞬間にデータモデルを起こす（stage2）。
- **壊れる条件に気づいて直す**。並行の競合を再現し、ロックで守る（stage3）。
- 各段で「なぜそう決めたか」を言葉にする。設計は結論より理由。

## 実装とテスト

`stage1_single_resource.py` / `stage2_multi_resource.py` / `stage3_concurrency_api.py` と、それぞれの `tests/`。段階を追って読むと、
単純な核が要件に応じて育っていく過程がわかる。各段は独立して実行・テストできる。
