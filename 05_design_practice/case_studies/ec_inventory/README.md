# ケーススタディ（ロードマップ）：EC在庫・注文

商品を在庫とともに扱い、顧客が複数商品をまとめて注文する EC の核を、ゼロから段階的に作る。
`design_method` の型を**ロードマップ形式**でたどる（各 Stage が型のどのステップを進めるか明記）。

予約システム（booking_system）と対になるケース。あちらの希少資源は「時間帯（連続した区間）」
だったが、EC の希少資源は「**個数（離散したカウント）**」。守る不変条件は「**在庫が負にならない**」。
そして booking が*処理・並行制御*が核だったのに対し、**EC はデータ設計が核**になる。

## design_method の型との対応

| 型のステップ | 主に扱う段階 |
|---|---|
| 1. 要件確認 | 各 Stage 冒頭の「要件／追加要件」 |
| 2. 機能洗い出し | 各 Stage の機能（buy / place_order / 一覧） |
| 3. データ設計 | **Stage 2（このケースの主役）** |
| 4. 構成 | Stage 3（API・エンドポイント） |
| 5. スケール対応 | Stage 3（トランザクション・制約による並行制御）／「さらに先」 |
| 6. 運用監視 | 「さらに先」節 |

---

## Stage 1：単一商品の在庫（一番単純な形）

*型: 要件確認 → 機能洗い出し → データ設計（最小）*

**要件（最小）**：1つの商品に在庫があり、購入で在庫を減らす。足りなければ売らない。

- 守る不変条件は「**在庫 >= 0**」。**減らす前に足りるか確認**し、足りなければ在庫を変えない。
- 割り切り：商品は1つ、単一スレッド前提。

→ `stage1.py`（`Inventory`）。まず動く核。ここが出発点。

## Stage 2：データ設計（★このケースの主役）

*型: 要件確認（追加）→ 機能洗い出し → **データ設計**（最も厚く作り込む）*

**追加要件**：商品は複数。顧客が複数商品を1回の注文でまとめて買う。注文の合計を出せる。

stage1 の「在庫カウント1つ」では表せない。ここで**エンティティ・関係・不変条件を厳密に**設計する。

- **エンティティと関係**
  - Customer 1 ─< Order（1対多）
  - Order 1 ─< OrderItem（1対多）、Product 1 ─< OrderItem（1対多）
  - `OrderItem` は Order と Product の多対多をつなぐ**属性を持つ中間エンティティ**。
    数量・単価という「関係の属性」を持つ（→ `01_data_layer/data_modeling`）。
- **不変条件（データに宿るルール）**
  - 在庫は負にならない。
  - 注文合計 = Σ(数量 × 単価)。合計は明細から**導出**する（重複して持たない）。
  - 単価は注文時点の価格を明細に**コピーして確定**（スナップショット）。後で値上げしても
    過去の注文金額は変わらない。
- **原子性の芽**：`place_order` は「全明細の在庫を確認してから減らす」。1つでも不足なら
  何も変えず失敗する（部分的に売れた状態を作らない）。stage3 でこれを DB のトランザクションに置き換える。

→ `stage2.py`（`Shop` と各 dataclass）。EC の学びの中心。

## Stage 3：実DB化・トランザクション・API

*型: **構成**（API）＋ **スケール対応**（DB のトランザクション・制約で並行制御）*

**追加要件**：永続化する。同時注文で最後の1個を取り合っても売り越さない。外部から使える API にする。

stage2 の「確認してから減らす」は単一スレッドでは正しいが、複数リクエストが割り込むと
両方が『在庫あり』と判断して**売り越す**（→ `04_reliability_and_scale/concurrency`）。
予約システムはプロセス内ロックで守ったが、**EC は DB のトランザクションと制約で守る**
（永続化もされ、複数サーバでも効く）。

- **条件付き減算**：`UPDATE products SET stock = stock - ? WHERE id = ? AND stock >= ?`。
  足りる行だけが更新される1文で、更新行数が0なら在庫不足。確認と減算の隙間が無い。
- **CHECK 制約**：`stock >= 0`。最後の砦。どの経路からも在庫を負にできない。
- **トランザクション**：注文全体（複数明細の減算＋注文＋明細の作成）を `with conn:` で囲み、
  途中の在庫不足で全ロールバック（原子性）。→ `01_data_layer/database_fundamentals`。
- **API**（→ `03_interface_layer/rest_api_design`）
  ```
  POST /orders          → 201 {order_id, total} / 409（在庫不足）/ 400
  GET  /products/{id}   → 200 {id,name,price,stock} / 404
  ```
  在庫不足は **409 Conflict** で返す。

→ `stage3.py`（`OrderDB` / `EcApp`）。在庫不足時に注文全体がロールバックし在庫が戻ること、
CHECK 制約が在庫を負にさせないことをテストで確認する。

## さらに先（型: スケール対応・運用監視）

- **並行制御の実測**：sqlite は書き込みを直列化する。規模が上がれば、行ロックや
  在庫を別テーブルに分けるなどで競合範囲を狭める。「在庫だけ Redis で持つ」構成もある。
- **予約在庫（カート）**：購入確定前に一定時間だけ在庫を押さえる（TTL 付きの hold。
  → `01_data_layer/caching` の TTL と同じ発想。予約システムの仮確保とも通じる）。
- **スケール**：商品IDでシャーディング（→ `scaling_patterns`）。ホット商品の在庫が単一障害点に
  なりやすい点に注意。
- **運用監視**：注文成功率・在庫不足（409）率・在庫切れ商品を計測し、発注や在庫配置を見直す。

## 予約システムとの対比（まとめ）

| | booking_system | ec_inventory |
|---|---|---|
| 希少資源 | 時間帯（区間） | 個数（カウント） |
| 核 | 区間の重なり・並行制御 | **データ設計**（関係・不変条件） |
| 並行制御 | プロセス内ロック | DB のトランザクション＋制約 |
| 主に効く層 | 02 interval / 04 concurrency | 01 data_modeling / database_fundamentals |

## 実装とテスト

`stage1.py` / `stage2.py` / `stage3.py` と各 `tests/`。段階を追うと、在庫カウント1つから
注文ドメインのデータ設計、そして実DBでの不変条件の守り方へと育っていく過程がわかる。
