# ケーススタディ：URL短縮サービス

長いURLを短いコードに変換し、そのコードへのアクセスを元URLへリダイレクトするサービス。
`design_method` の型（要件確認 → 機能洗い出し → データ設計 → 構成 → スケール対応 → 運用監視）
に沿って設計する。

## 1. 要件確認

- **目的**：長いURLを短くし、短縮URLへのアクセスを元URLへ転送する。
- **スコープ（今回やる）**：短縮コードの発行、元URLへの解決、アクセス数の計測。
- **今回やらない**：ユーザー認証、カスタムドメイン、UI。
- **規模感・特性**：**読み取り（解決）が書き込み（発行）より圧倒的に多い**。
  一度作られた短縮URLは大量にアクセスされうる。→ 読み取り経路の最適化が要になる。
- **一貫性**：発行済みコードは常に同じURLへ解決される必要がある（強い一貫性）。
  クリック数は多少の遅延・誤差を許容してよい。

## 2. 機能洗い出し

- **shorten(long_url) → code**（Must）：元URLを登録して短縮コードを返す。
- **resolve(code) → long_url**（Must）：コードから元URLを引く。中心の機能。
- **clicks(code) → 回数**（Nice）：アクセス数の計測。運用監視に使う。
- 拡張余地：カスタム別名、有効期限、重複URLの共有。

まず中心の shorten / resolve から固める。

## 3. データ設計

中心はごく単純な対応関係。「コード → 元URL」を引ければよい。

```
url_mapping
- code (主キー)          # 短縮コード（例: "b7Fk"）
- long_url               # 元URL
- created_at
- click_count            # 計測（運用監視）
```

**コードの作り方が設計判断の肝**。本実装は「連番ID を base62 で符号化」する。

- 連番＋base62 は **衝突しない・短い・確定的**。6文字で約568億通り。
- トレードオフ：連番は**推測・列挙されやすい**（隣のコードを試せる）。秘匿が要るなら
  乱数コード＋衝突チェックにする。要件（今回は公開URL）に照らして連番で十分と判断。
- 別解：元URLをハッシュ → 同じURLが同じコードになる（重複排除）が、衝突対策が要る。

（→ `01_data_layer/data_modeling` / `database_fundamentals`。code に一意制約を張る。）

## 4. 構成

REST で2つの経路を用意する（→ `03_interface_layer/rest_api_design`）。

```
POST /api/urls       body: {long_url}      → 201 {code}
GET  /{code}                                → 302 リダイレクト（元URLへ）
```

- **書き込み経路**：`POST /api/urls` → 採番 → 保存。
- **読み取り経路**：`GET /{code}` → 解決 → リダイレクト。ここが最も呼ばれる。

**パスの衝突に注意**：リダイレクトは root 直下の catch-all `/{code}`。ここに管理APIを
`/urls` で置くと、`/urls` がコード `"urls"` と誤解される。そこで管理APIは **`/api/` 配下**に
分ける。root の1階層を短縮コード専用に空けておくのがこの種のサービスの定石。

実装は2段構え。`ShortenerService`（`shorten` / `lookup` / `record_click`）がドメインの核。
`app.py` の `UrlShortenerApp` が、その周りに 03 のルータ・01 のキャッシュ・04 のレート制限を
実際に配線した**動く API**。

## 5. スケール対応

読み取りが極端に多いサービスなので、解決経路を優先して最適化する（→ `04_reliability_and_scale`）。
このうちキャッシュとレート制限は `app.py` で**実際に配線済み**。

- **ホットURLのキャッシュ**（実装済み）：`GET /{code}` の前段に `SimpleCache` を Cache-Aside で
  挟む。code → long_url は不変なのでキャッシュ向き（→ `01_data_layer/caching`）。
  クリック計測はキャッシュヒットでも数えるよう、読み取り（lookup）と計測（record_click）を分けている。
- **発行のレート制限**（実装済み）：`POST /api/urls` を `TokenBucketLimiter` でクライアント単位に
  制限し、超過は 429 を返す（→ `rate_limiting`）。
- **データ分割（シャーディング）**（解説）：コードをキーに複数ノードへ分散。ノード増減で再配置を
  抑えるためコンシステントハッシュを使う（→ `scaling_patterns`）。単一プロセスでは実演しにくいため
  ここでは設計方針のみ。
- **採番の分散**（解説）：単一の連番採番は書き込みの単一障害点になりうる。ノードごとに
  ID レンジを配る等で分散できる（トレードオフ：コードが完全な連番ではなくなる）。

## 6. 運用監視

- **計測**：コードごとのクリック数（`clicks`）、発行数、解決の成功/失敗（404）率。
- **異常検知**：特定コードへの異常なアクセス集中、404 の急増。
- **調整**：キャッシュ TTL やレート上限を、計測値を見て見直す。
- クリック数は書き込みが頻繁なので、即時 DB 反映ではなく非同期集計にする判断もある
  （正確さより負荷軽減を優先。→ キャッシュ／キューの考え方）。

## 実装

- `url_shortener.py` — ドメインの核。`encode_base62` / `decode_base62`（コードの採番）、
  `ShortenerService`（`shorten` / `lookup` / `record_click` / `resolve`）。
- `app.py` — 他レイヤーを配線した動く API。`UrlShortenerApp` が
  **03 の `Router`**（ルーティング）・**01 の `SimpleCache`**（解決経路の Cache-Aside）・
  **04 の `TokenBucketLimiter`**（発行のレート制限）を実際に import して組み合わせる。
  `handle(method, path, body)` に投げると `(status, body)` が返る。

`tests/` で、201/302/404/405/429 の応答、キャッシュが効いていること（核のストアから消しても
キャッシュから解決できる）、キャッシュヒットでもクリックが数えられることを確認している。

（層をまたぐ import は、ディレクトリが数字始まりのため `app.py` 内で `sys.path` に各モジュールの
場所を足して行っている。）
