# 設計の進め方（型）

サービス設計を、毎回同じ順序で進めるための「型」。
思いつきで実装に飛びつかず、**要件確認 → 機能洗い出し → データ設計 → 構成 → スケール対応 → 運用監視**
の順にたどることで、抜け漏れを防ぎ、判断の理由を言葉にしながら進められる。

case_studies の各ケースは、この型の見出しに沿って設計を記述する。

## 1. 要件確認

「何を作るか」を、曖昧なまま進めない。まず前提を確認して固める。

- **目的とスコープ**：何を解決する？ 今回やらないことは？
- **利用者と規模感**：誰が使う？ 想定リクエスト数・データ量は？ 読み多い／書き多い？
- **制約**：一貫性はどこまで必要？ 遅延の許容は？ 使える資源は？

不明点はここで質問して潰す。**要件が曖昧なまま設計すると、後で全部やり直しになる。**

## 2. 機能洗い出し

要件を、具体的な機能（ユースケース）に分解する。名詞（登場するモノ）と
動詞（それに対する操作）で挙げると、後のデータ設計・API 設計に直結する。

- 主要機能（Must）と、あると良い機能（Nice to have）を分ける
- **まず簡単なものから**。中心の1機能を先に決め、そこから広げる

## 3. データ設計

機能が扱うデータを、エンティティ・関係・属性に落とす（→ `01_data_layer/data_modeling`）。
**ここが設計の心臓部**。データ構造が素直なら、多くの機能は自然に実装できる。

- エンティティと 1対多／多対多、属性を持つ中間テーブルはないか
- 主キー・外部キー・一意制約（→ `01_data_layer/database_fundamentals`）
- 時点の事実として確定保存すべき値はないか（スナップショット）

## 4. 構成

データに対する操作を、外部インターフェースとして設計する。

- リソースとエンドポイント（→ `03_interface_layer/rest_api_design`）
- 主要な処理の流れ（読み取り経路・書き込み経路）
- どこにキャッシュを挟むか（→ `01_data_layer/caching`）

まずは**単一構成の動く形**を描く。分散はこの後の段で足す。

## 5. スケール対応

「規模が増えたらどこが詰まるか」を予測し、打ち手を当てる（→ `04_reliability_and_scale`）。
ただし**最初から作り込まない**。ボトルネックを見極めてから打つ。

- 読み取りが多い → キャッシュ、リードレプリカ
- データが1台に乗らない → シャーディング（コンシステントハッシュ）
- 同時更新の競合 → ロック／トランザクション
- 過剰なアクセス → レート制限
- 重い処理 → メッセージキューで非同期化

## 6. 運用監視

作って終わりではない。動かし続けるために、状態を計測できるようにする。

- 何を記録するか（リクエスト数、エラー率、遅延、利用状況）
- 異常にどう気づくか（閾値・アラート）
- 設定値（キャッシュ TTL、レート上限など）を、計測を見て見直す

## この型の使い方

- **上から順に、ただし行き来してよい**。スケールを考えてデータ設計に戻る、はよくある。
- **各段で「なぜそう決めたか」を言葉にする**。設計は結論より理由が大事。
- **簡単な中心から広げる**。全部を一度に完璧にせず、動く核を作ってから拡張する。

（この型は進め方＝プロセスなので、実装コードは持たない。適用例は `case_studies` を参照。）
